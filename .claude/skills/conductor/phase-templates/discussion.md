# Phase 1: DISCUSSION / DESIGN（发散讨论）

## 目的
在方向不明确时，通过多视角碰撞确定方向、技术方案或设计方案。

## 触发条件
仅 COMPLEX 任务进入此 Phase。

## 两种子模式

### 模式 A: Design First（涉及 UI/前端）

**何时使用**：任务涉及新页面、新组件或交互变更。

**执行步骤**：
1. Spawn `rams-designer`，传入：
   - 任务描述
   - 现有界面/组件信息（如有）
   - 用户场景和约束
2. 收到设计方案后，评估方案完整性
3. 将设计方案作为后续 Phase 的约束条件

**传给 rams-designer 的 prompt 模板**：
```
请为以下功能出设计方案：

任务：[任务描述]
现有界面：[描述或"无，全新功能"]
目标用户：[描述]
约束条件：[技术约束、平台约束等]

请产出完整的设计方案，包含交互流程、视觉规范、组件清单。
```

### 模式 B: Multi-Perspective Discussion（需要方向碰撞）

**何时使用**：技术方案有多种选择，需要讨论确定方向。

**执行步骤**：
1. 并行 spawn ≥2 个 agent（选择有不同视角的 persona）：
   - `jobs-planner` — 产品/用户体验视角
   - `torvalds-planner` — 架构/技术视角
   - `rams-planner` — 设计/交互视角（如涉及 UI）
2. 给每个 agent 相同的议题，但引导不同的关注点
3. 收集各方分析结果
4. **Conductor 综合决策**，产出方向性结论

**传给 discussion agent 的 prompt 模板**：
```
请从你的专业角度分析以下任务：

任务：[任务描述]
背景：[项目背景、现有代码概况]
需要讨论的核心问题：[列出 1-3 个关键问题]

请给出你的分析和建议，包含：
1. 你认为最佳方案是什么？
2. 关键风险和注意事项
3. 你不同意的方案及理由
```

## 输出

Conductor 需要产出一个 **Discussion Summary**，包含：
1. **核心决策**：确定了什么方向
2. **各方观点**：每个 agent 的关键观点
3. **争议点及决策**：有分歧的地方，最终选了什么，为什么
4. **对 Phase 2 的约束**：这些决策对后续 planning 的具体影响

## 失败处理
- 如果各方意见完全对立无法收敛 → Conductor 根据技术合理性裁决
- 如果 Design 方案不完整 → 给 rams-designer 反馈，要求补充
- 最多迭代 2 次 → 超过则用现有最佳方案进入 Phase 2
