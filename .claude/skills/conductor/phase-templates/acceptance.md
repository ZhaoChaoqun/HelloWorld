# Phase 5: ACCEPTANCE（验收）

## 目的
对照原始需求和验收标准，做最终的产品级验收。

## 执行步骤

### Step 1: Spawn Acceptance Reviewer

- spawn `jobs-reviewer` — 从产品/用户体验角度做最终验收

**传给 jobs-reviewer 的 prompt 模板**：
```
请对以下开发成果做最终验收：

## 原始任务
[完整的任务描述]

## 执行计划（参考）
[ExecutionPlan 摘要]

## 验收标准
[来自 ExecutionPlan 的验收标准列表]

## 实际完成情况
[各步骤的完成报告汇总]

## Quality Gate 结果
[review 和 test 的最终结果]

## 验收要求
请逐条检查验收标准：
1. 功能是否完整实现？
2. 用户体验是否达标？
3. 有没有遗漏的需求点？
4. 整体品质是否达到"让用户尖叫"的标准？

产出验收报告：PASS / FAIL + 逐条检查结果
```

### Step 2: 判断结果

| 场景 | 行动 |
|------|------|
| PASS | → 🎉 任务完成！产出最终报告 |
| FAIL（小问题） | → iterations++，回到 Phase 3 修改特定步骤 |
| FAIL（方向问题） | → iterations++，回到 Phase 2 重新规划 |
| FAIL（设计问题） | → iterations++，回到 Phase 1 重新设计 |
| 超过 3 次迭代 | → FAILED，通知用户 |

### Step 3: 最终报告

**任务完成报告格式**：
```
## ✅ 任务完成报告

### 任务
[任务描述]

### 执行摘要
- 复杂度判断：TRIVIAL / SMALL / COMPLEX
- 经历阶段：Phase 0 → ... → Phase 5
- 总迭代次数：X

### 关键决策
1. [决策 1]：[选了什么] — [为什么]
2. [决策 2]：...

### 交付物
- [文件/功能 1]：[描述]
- [文件/功能 2]：[描述]

### 质量验证
- Code Review：[评分] (by [reviewer])
- Testing：[结果] (by beck-tester)
- Acceptance：[结果] (by jobs-reviewer)

### 后续建议（如有）
- [建议 1]
- [建议 2]
```

**任务失败报告格式**：
```
## ❌ 任务未完成报告

### 任务
[任务描述]

### 执行摘要
- 复杂度判断：[...]
- 停止阶段：Phase X
- 总迭代次数：X（达到上限）

### 问题分析
1. [核心问题]：[描述]
2. [阻塞原因]：[描述]

### 已完成部分
- [已完成的步骤和交付物]

### 建议
- [如何继续推进的建议]
- [需要用户做什么决策]
```

## 迭代规则

- 每次 FAIL 返回时，iterations 计数器 +1
- 最大迭代次数：3
- 返回的目标 Phase 由失败原因决定：
  - 代码 bug → Phase 3（重新执行对应步骤）
  - 方案不合理 → Phase 2（重新规划）
  - 需求理解错误 → Phase 1（重新讨论）或通知用户
